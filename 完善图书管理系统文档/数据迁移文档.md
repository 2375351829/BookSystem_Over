# 数据迁移文档

## 1. 概述

### 1.1 数据库版本管理工具介绍

本系统使用**自定义迁移脚本方式**进行数据库版本管理，而非Flyway或Liquibase等自动化工具。迁移脚本存放在 `src/main/resources/db/migration/` 目录下。

#### 迁移策略说明：
- **命名规范**：采用类似Flyway的 `V{version}__{description}.sql` 格式
- **版本号规则**：使用语义化版本号（如 V1.0.1, V1.0.2）
- **执行顺序**：按版本号从小到大依次执行
- **回滚支持**：每个迁移脚本都有对应的 `_rollback.sql` 回滚脚本
- **幂等性设计**：部分脚本包含存在性检查，可重复执行

#### 为什么不使用Flyway/Liquibase：
1. 本项目处于开发阶段，需要灵活的手动控制
2. 部分迁移涉及复杂的数据转换逻辑
3. 便于调试和问题排查
4. 降低学习成本

### 1.2 迁移脚本存放位置和命名规范

#### 幂等性设计

所有字段添加都使用了动态SQL检查是否已存在：

```sql
SET @preparedStatement = (SELECT IF(
  (SELECT COUNT(*) FROM INFORMATION_SCHEMA.COLUMNS
   WHERE TABLE_SCHEMA = @dbname AND TABLE_NAME = @tablename AND COLUMN_NAME = @columnname) > 0,
  'SELECT 1',  -- 已存在则跳过
  CONCAT('ALTER TABLE ', @tablename, ' ADD COLUMN ', ...)  -- 不存在则添加
));
```

**优点**：
- 可重复执行而不会报错
- 适用于多次部署场景
- 安全可靠

```sql

DELIMITER $$
CREATE PROCEDURE add_foreign_key_if_not_exists(
    IN p_table_name VARCHAR(100),
    IN p_constraint_name VARCHAR(100),
    IN p_column_name VARCHAR(100),
    IN p_ref_table VARCHAR(100),
    IN p_ref_column VARCHAR(100),
    IN p_on_delete VARCHAR(20),
    IN p_on_update VARCHAR(20)
)
BEGIN
    DECLARE fk_count INT DEFAULT 0;

    SELECT COUNT(*) INTO fk_count
    FROM information_schema.TABLE_CONSTRAINTS
    WHERE TABLE_SCHEMA = DATABASE()
      AND TABLE_NAME = p_table_name
      AND CONSTRAINT_NAME = p_constraint_name
      AND CONSTRAINT_TYPE = 'FOREIGN KEY';

    IF fk_count = 0 THEN
        -- 执行添加外键
    ELSE
        -- 外键已存在，跳过
    END IF;
END$$
DELIMITER ;
```

#### 索引类型和用途详解

**组合索引（Composite Index）**：
- `idx_user_status` (notification): 加速"查询某用户的未读通知"场景
- `uk_room_seat` (seat): **唯一索引**，确保同一阅览室内座位编号不重复
- `idx_room_date` (seat_reservation): 加速"查询某阅览室某天的预约情况"

**单列索引（Single Column Index）**：
- 时间字段索引：用于ORDER BY排序优化
- 状态字段索引：用于WHERE条件过滤
- 类型字段索引：用于分类筛选

#### 性能优化建议

这些索引针对以下高频查询场景优化：
1. **通知模块**：用户查看未读通知列表
2. **座位管理**：查询可用座位、按类型筛选
3. **预约系统**：查询某天的预约情况
4. **采购模块**：检查ISBN是否已申请过

## 3. 新迁移脚本编写规范

### 3.1 命名规则

**格式**：`V{version}__{description}.sql`

**示例**：
```
V1.0.9__add_payment_table.sql
V1.1.0__upgrade_user_role_system.sql
V2.0.0__refactor_borrow_flow.sql
```

**规则**：
- 版本号使用点分格式：主版本.次版本.修订号
- 版本号后跟双下划线 `__`
- 描述使用小写字母和下划线
- 描述应简洁明了地说明变更内容

### 3.2 版本号递增规则

- **补丁版本**（修复bug）：递增第三位，如 1.0.8 -> 1.0.9
- **次版本**（新增功能）：递增第二位，如 1.0.9 -> 1.1.0
- **主版本**（重大变更）：递增第一位，如 1.9.9 -> 2.0.0

**当前最新版本**：V1.0.8
**下一个版本建议**：V1.0.9

### 3.3 DDL语句编写规范

#### 文件头模板

```sql
-- ============================================
-- 数据库迁移脚本: V{version}
-- 功能: {简短描述}
-- 作者: {作者名}
-- 日期: {YYYY-MM-DD}
-- 数据库: CityLibrary
-- 字符集: utf8mb4
-- ============================================

-- 使用事务确保原子性
START TRANSACTION;

-- ============================================
-- 变更内容
-- ============================================

-- 提交事务
COMMIT;

-- ============================================
-- 迁移脚本执行完成
-- 变更统计: ...
-- ============================================
```

#### 编写要求

1. **使用事务包裹**：确保原子性
2. **添加注释**：每个变更都要注释说明
3. **使用IF EXISTS/IF NOT EXISTS**：提高幂等性
4. **指定字符集**：统一使用utf8mb4
5. **添加字段COMMENT**：便于理解用途

### 3.4 回滚脚本编写规范

**命名**：在原文件名后加 `_rollback`

**示例**：
```
V1.0.9__add_payment_table.sql
V1.0.9__add_payment_table_rollback.sql
```

**回滚脚本内容原则**：
- DDL操作的反向操作（CREATE <-> DROP, ADD <-> DROP）
- ⚠️ 数据操作（UPDATE/DELETE）通常不提供回滚
- 注明限制和警告

**回滚脚本模板**：

```sql
-- ============================================
-- 数据库回滚脚本: V{version}
-- 功能: 回滚 {描述}
-- 注意: 此脚本的局限性说明
-- ============================================

START TRANSACTION;

-- 回滚操作...

COMMIT;

SELECT '回滚脚本执行完成！' AS '执行结果';
```

### 3.5 注意事项

#### ✅ 必须遵守的原则

1. **不可修改已执行的迁移**
   - 已部署到生产环境的脚本不能修改
   - 如需修复，必须创建新的迁移脚本

2. **保证幂等性**
   - 脚本可重复执行而不出错
   - 使用 IF NOT EXISTS / INFORMATION_SCHEMA 检查

3. **小步快跑**
   - 每个迁移脚本只做一件事
   - 避免在一个脚本中混合多种类型的变更

4. **测试环境先验证**
   - 先在开发/测试环境执行
   - 确认无误后再应用到生产

5. **备份后再迁移**
   - 执行重要迁移前备份数据库
   - 保留回滚能力

#### ❌ 禁止的操作

1. **不要在生产环境直接执行DDL**
   - 必须先经过测试环境验证
   - 必须有回滚方案

2. **不要在迁移中操作大量数据**
   - 大数据量更新应在应用层分批处理
   - 避免锁表时间过长

3. **不要删除字段或表（除非必要）**
   - 优先标记为废弃（deprecated）
   - 确认无依赖后再物理删除

4. **不要使用SELECT ***
   - 明确指定需要的字段
   - 避免意外的结构变化影响

   ```python
   import pandas as pd
   import pymysql
   from bcrypt import hashpw, gensalt

   # 读取旧数据
   old_users = pd.read_sql("SELECT * FROM user", old_conn)

   # 数据转换
   new_users = []
   for _, row in old_users.iterrows():
       new_user = {
           'username': row['username'],
           'password': hashpw(row['password'].encode(), gensalt()).decode(),
           'realName': row['real_name'],
           'userType': map_user_type(row['type']),
           'deleted': 0,
           'createTime': pd.Timestamp.now(),
           'updateTime': pd.Timestamp.now()
       }
       new_users.append(new_user)

   pd.DataFrame(new_users).to_sql('user_account', new_conn, if_exists='append', index=False)
   ```

   **方案B：使用存储过程**

   ```sql
   DELIMITER $$

   CREATE PROCEDURE migrate_users()
   BEGIN
       DECLARE done INT DEFAULT FALSE;
       DECLARE old_id, new_id BIGINT;
       DECLARE cur CURSOR FOR SELECT id FROM old_system.user;
       DECLARE CONTINUE HANDLER FOR NOT FOUND SET done = TRUE;

       OPEN cur;

       read_loop: LOOP
           FETCH cur INTO old_id;
           IF done THEN
               LEAVE read_loop;
           END IF;

           -- 插入到新表
           INSERT INTO user_account (username, password, ...)
           SELECT username, ENCRYPT_PASSWORD(password), ...
           FROM old_system.user WHERE id = old_id;

           SET new_id = LAST_INSERT_ID();

           -- 记录ID映射关系
           INSERT INTO id_mapping (old_table, old_id, new_table, new_id)
           VALUES ('user', old_id, 'user_account', new_id);
       END LOOP;

       CLOSE cur;
   END$$

   DELIMITER ;
   ```

#### 常见数据质量问题及解决方案

| 问题类型 | 示例 | 解决方案 |
|---------|------|---------|

### 6.6 迁移回滚方案

#### 回滚触发条件

出现以下情况时应考虑回滚：
- 数据丢失超过阈值（如>0.1%）
- 核心业务功能异常
- 性能严重下降（如响应时间>10秒）
- 数据一致性问题无法快速修复

#### 回滚步骤

##### 方案A：快速回滚（如果有完整备份）

```bash

systemctl stop citylibrary-backend

# 2. 恢复数据库备份
mysql -u root -p CityLibrary < backup_before_migration.sql

# 3. 重启旧系统（如果还在运行）
# 或启动新系统（使用旧数据）
systemctl start citylibrary-backend

# 4. 验证服务恢复
curl http://localhost:8080/api/health
```

##### 方案B：部分回滚（仅回滚某些表的迁移）

```sql
-- 1. 清空新导入的数据（按反向依赖顺序）
SET FOREIGN_KEY_CHECKS = 0;

TRUNCATE TABLE borrow_record;
TRUNCATE TABLE book;
TRUNCATE TABLE user_account;

SET FOREIGN_KEY_CHECKS = 1;

-- 2. 从备份恢复这些表
mysql -u root -p CityLibrary < partial_backup.sql

-- 3. 重新执行迁移（修复问题后）
```

##### 方案C：数据修复（不回滚，修复问题数据）

```sql
-- 示例：补充缺失的用户
INSERT INTO user_account (username, password, ...)
SELECT username, ENCRYPT_PASSWORD(password), ...
FROM old_system.user
WHERE id NOT IN (SELECT old_id FROM id_mapping WHERE old_table = 'user');

-- 示例：修复断裂的外键引用
UPDATE borrow_record br
JOIN id_mapping im ON br.old_user_id = im.old_id
SET br.userId = im.new_id
WHERE br.userId NOT IN (SELECT id FROM user_account);
```

### 7.3 小步快跑，每次迁移尽量小

**重要性**：⭐⭐⭐⭐

**原则**：

1. **单一职责**
   - 每个迁移脚本只做一个事情
   - 如：只添加字段、只创建表、只修改数据

2. **版本粒度适中**
   - ✅ 好：V1.0.7添加字段，V1.0.8迁移数据
   - ❌ 差：V1.0.7同时添加字段、创建索引、迁移数据

3. **可独立回滚**
   - 每个版本都能单独回滚
   - 回滚不影响其他版本的变更

4. **频繁提交**
   - 完成一个功能点就创建新版本
   - 不要积累大量变更一次性发布

**示例对比**：

```sql
-- ❌ 错误：大而全的迁移脚本

-- ✅ 正确：拆分为多个小迁移
-- V1.0.9__add_new_table_a.sql
-- V1.0.10__add_new_table_b.sql
-- V1.0.11__modify_field_x.sql
-- V1.0.12__migrate_data_to_new_structure.sql
-- V1.0.13__optimize_indexes.sql
```

   ```sql
   --
   -- ⚠️ 重要提醒：
   -- 1. 此脚本会删除V1.0.2创建的所有表及其数据
   -- 2. 如果这些表已被后续迁移引用，需要先回滚后续版本
   -- 3. 删除操作不可逆，请确认已备份
   --
   ```

4. **考虑级联回滚**

   ```sql
   -- 回滚顺序（反向拓扑排序）
   -- 1. 先删除有外键引用的表
   DROP TABLE IF EXISTS book_review;
   DROP TABLE IF EXISTS book_import_detail;
   DROP TABLE IF EXISTS user_behavior;
   DROP TABLE IF EXISTS book_recommendation;
   -- 2. 再删除被引用的表（如果有新增的）
   ```

### 7.5 记录迁移日志

**重要性**：⭐⭐⭐

**日志记录最佳实践**：

1. **数据库迁移日志表**

   ```sql
   CREATE TABLE migration_log (
       id BIGINT AUTO_INCREMENT PRIMARY KEY,
       version VARCHAR(20) NOT NULL,
       description VARCHAR(200),
       executed_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
       execution_time_ms INT,
       status ENUM('SUCCESS', 'FAILED', 'ROLLED_BACK'),
       error_message TEXT,

   );
   ```

2. **每个迁移脚本内部记录**

   ```sql
   UPDATE migration_log
   SET status = 'SUCCESS',
   ```

#### 幂等性检查模式

```sql
-- 检查字段是否存在
SET @sql = IF(
  (SELECT COUNT(*) FROM INFORMATION_SCHEMA.COLUMNS
   WHERE TABLE_SCHEMA = DATABASE() AND TABLE_NAME = 'table' AND COLUMN_NAME = 'col') > 0,
  'SELECT "Column exists"',
  'ALTER TABLE table ADD COLUMN col ...'
);

-- 检查外键是否存在
SET @sql = IF(
  (SELECT COUNT(*) FROM INFORMATION_SCHEMA.TABLE_CONSTRAINTS
   WHERE CONSTRAINT_NAME = 'fk_name') > 0,
  'SELECT "FK exists"',
  'ALTER TABLE ADD CONSTRAINT fk_name ...'
);

-- 检查索引是否存在
SET @sql = IF(
  (SELECT COUNT(*) FROM INFORMATION_SCHEMA.STATISTICS
   WHERE INDEX_NAME = 'idx_name') > 0,
  'SELECT "Index exists"',
  'CREATE INDEX idx_name ...'
);
```

### B. 故障排查指南

#### 常见错误及解决方案

| 错误信息 | 原因 | 解决方案 |
|---------|------|---------|
| Duplicate entry for key 'unique_key' | 违反唯一约束 | 检查数据去重或修改约束 |
| Cannot add foreign key constraint | 外键引用不存在 | 先创建被引用的表/字段 |
| Unknown column 'xxx' | 字段不存在 | 先执行添加字段的迁移 |
| Table doesn't exist | 表不存在 | 先执行建表迁移 |
| Lock wait timeout | 表锁等待超时 | 在低峰期执行或杀掉阻塞查询 |
| Out of range value | 值超出字段范围 | 检查数据类型或扩大字段长度 |
