## 2. 项目概述

### 2.1 项目简介
城市图书馆管理系统（CityLibrary System）是一个功能完善的图书馆管理平台，采用前后端分离架构，支持图书借阅、座位预约、咨询问答、数据统计等核心业务功能。

### 2.2 技术架构

#### 后端技术栈
- **框架**: Spring Boot 3.2.0
- **Java版本**: JDK 17
- **ORM框架**: MyBatis-Plus 3.5.7
- **数据库**: MySQL 8.0 (数据库名: CityLibrary)
- **安全框架**: Spring Security + JWT (jjwt 0.12.5)
- **构建工具**: Maven
- **其他依赖**:
  - Lombok 1.18.30 (代码简化)
  - Apache POI 5.2.3 (Excel导入导出)
  - iTextPDF 5.5.13.3 (PDF生成)
  - Spring Validation (参数校验)

#### 前端技术栈
- **框架**: Vue.js 3.5.30
- **构建工具**: Vite 8.0.1
- **语言**: TypeScript ~5.9.3
- **UI组件库**: Element Plus 2.13.6
- **状态管理**: Pinia 3.0.4
- **路由**: Vue Router 4.6.4
- **国际化**: vue-i18n 12.0.0-alpha.3
- **图表库**: ECharts 5.6.0
- **HTTP客户端**: Axios 1.14.0
- **Excel处理**: xlsx 0.18.5
- **样式预处理**: Sass 1.80.8
- **进度条**: nprogress 0.2.0

### 2.3 主要模块划分

| 模块 | 功能描述 |
|------|----------|
| **用户认证模块** | 用户注册、登录、JWT令牌管理、角色权限控制 |
| **图书管理模块** | 图书CRUD、批量导入、分类管理、ISBN查询 |
| **借阅管理模块** | 借书、还书、续借、逾期检查、罚款计算 |
| **预约模块** | 图书预约、座位预约、超时自动释放 |
| **用户中心模块** | 个人信息、收藏、阅读历史、消息通知 |
| **咨询模块** | 用户提问、管理员回复、FAQ管理 |
| **采购模块** | 图书采购申请、采编任务管理 |
| **统计报表模块** | 借阅统计、热门排行、用户行为分析 |
| **系统配置模块** | 借阅规则、系统参数、操作日志 |
| **数据导出模块** | Excel/PDF格式数据导出 |

## 7. 代码组织规范

### 7.1 命名约定

#### 7.1.1 Java类命名规范（后端）

| 类型 | 命名规则 | 示例 |
|------|----------|------|
| Controller | `{模块名}Controller` | `BookController`, `UserController` |
| Service | `{模块名}Service` | `BookService`, `UserService` |
| Model/Entity | `{业务名词}` (大驼峰) | `UserAccount`, `BorrowRecord` |
| Mapper/Repository | `{实体名}Mapper` | `UserAccountMapper`, `BookMapper` |
| Config | `{功能}Config` | `SecurityConfig`, `WebConfig` |
| Exception | `{异常场景}Exception` | `UserNotFoundException` |
| Task | `{任务名}Task` | `OverdueCheckTask` |
| Util/Tool | `{功能名}Validator/Util` | `PasswordValidator` |
| Constant | `{领域}Constants` | `StatusConstants` |
| Enum | `{枚举名}` (单数) | `UserType`, `FineStatus` |

**特点**:
- 严格遵循大驼峰命名法（PascalCase）
- 类名具有明确的业务语义
- 避免缩写，保持完整性
- Controller/Service/Repository三层对应同一业务模块

#### 7.1.2 TypeScript/Vue文件命名规范（前端）

| 类型 | 命名规则 | 示例 |
|------|----------|------|
| Vue组件（页面） | `{功能名}View.vue` | `HomeView.vue`, `BookDetailView.vue` |
| Vue组件（管理页） | `Admin{功能名}View.vue` | `AdminBookView.vue`, `AdminUserView.vue` |
| Vue组件（公共） | `{功能名}.vue` (PascalCase) | `BookCard.vue`, `StatCard.vue` |
| API模块 | `{模块名}.ts` (camelCase) | `auth.ts`, `bookReview.ts` |
| Store | `{领域名}.ts` (camelCase) | `authStore.ts`, `user.ts` |
| 路由 | `index.ts`, `guard.ts` | 固定命名 |
| 配置 | `{功能名}.ts` | `adminMenu.ts` |
| 样式 | `{用途}.scss` | `variables.scss`, `global.scss` |
| 工具函数 | `{功能名}.ts` | `request.ts` |
| 类型定义 | `types.ts` | 固定命名 |

**特点**:
- Vue组件使用PascalCase（与Vue官方推荐一致）
- 页面组件统一加`View`后缀区分
- 管理后台页面加`Admin`前缀
- JS/TS文件使用camelCase
- API和Store与后端模块一一对应

#### 7.1.3 数据库表和字段命名规范

**表命名规则**:
- 全小写字母
- 单词间用下划线分隔（snake_case）
- 使用单数形式（非复数）
- 示例: `user_account`, `borrow_record`, `book_import_batch`

**字段命名规则**:
- 全小写字母
- snake_case风格
- 主键统一命名为 `id` (BIGINT AUTO_INCREMENT)
- 外键命名为 `{关联表}_id` (如 `user_id`, `book_id`)
- 时间戳字段: `create_time`, `update_time`
- 逻辑删除字段: `deleted` (TINYINT 0/1)
- 状态字段: `status` (根据业务定义值)
- 示例: `student_id`, `faculty_id`, `due_date`, `renew_count`

**索引命名规则**:
- 主键: `PRIMARY KEY`
- 唯一索引: `uk_{字段名}` (如 `uk_username`)
- 普通索引: `idx_{字段名}` (如 `idx_title`, `idx_category`)

**特点**:
- 与MyBatis-Plus的`map-underscore-to-camel-case`配置配合
- Java实体属性自动转换为camelCase（如 `user_id` → `userId`）
- 保持命名的一致性和可读性

#### 7.1.4 API路径命名规范

**RESTful API设计原则**:

| HTTP方法 | 路径模式 | 说明 | 示例 |
|----------|----------|------|------|
| GET | `/api/{resources}` | 获取列表 | `GET /api/books` |
| GET | `/api/{resources}/{id}` | 获取单个 | `GET /api/books/123` |
| POST | `/api/{resources}` | 创建资源 | `POST /api/books` |
| PUT | `/api/{resources}/{id}` | 更新资源 | `PUT /api/books/123` |
| DELETE | `/api/{resources}/{id}` | 删除资源 | `DELETE /api/books/123` |

**命名规范**:
- 全部小写
- 使用复数名词表示资源集合
- 使用连字符(-)连接复合词（非下划线）
- 动作性接口使用动词或`/actions`子路径
- 示例:
  - `/api/books` - 图书资源
  - `/api/user-account` - 用户账户（复合词）
  - `/api/borrows/123/return` - 还书动作
  - `/api/borrows/123/renew` - 续借动作

**查询参数**:
- 分页: `?page=1&size=10`
- 排序: `?sort=create_time&order=desc`
- 搜索: `?keyword=java&category=tech`
- 过滤: `?status=available&year=2024`

### 7.2 文件划分原则

#### 7.2.1 Controller与Service对应关系

**一对一映射原则**:
每个Controller通常对应一个同名的Service，职责清晰：

```
BookController.java  ←→  BookService.java
UserController.java  ←→  UserService.java  
BorrowController.java ←→  BorrowService.java
...
```

**例外情况**:
- `AuthController` → 使用 `UserService` + `UserDetailsService` + `RefreshTokenService`
- `TestController` → 直接返回简单测试数据，无对应Service
- `HealthController` → 仅做健康检查，调用多个Service

**调用链路**:
```
HTTP Request
    ↓
Controller (接收参数、参数校验、调用Service)
    ↓
Service (业务逻辑、事务管理、调用Mapper)
    ↓
Mapper/Repository (SQL执行、数据返回)
    ↓
Database
```

#### 7.2.2 页面与路由对应关系

**路由组织原则**:
- 用户端页面作为DefaultLayout的子路由
- 管理后台页面作为AdminLayout的子路由
- 路由路径与功能语义一致

**页面命名与路由映射**:
```
/views/HomeView.vue          →  /
/views/BookListView.vue      →  /books
/views/BookDetailView.vue    →  /books/:id
/views/admin/AdminBookView.vue → /admin/books
```

**懒加载策略**:
所有路由组件使用动态导入(`() => import(...)`)，实现按需加载，优化首屏性能。

#### 7.2.3 API模块与后端模块对应关系

**完全对应原则**:
前端`src/api/`下的每个文件对应后端的一个Controller：

| 前端API文件 | 后端Controller | 业务模块 |
|-------------|----------------|----------|
| `api/auth.ts` | `AuthController` | 认证授权 |
| `api/user.ts` | `UserController` | 用户管理 |
| `api/book.ts` | `BookController` | 图书管理 |
| `api/borrow.ts` | `BorrowController` | 借阅管理 |
| `api/category.ts` | `CategoryController` | 分类管理 |
| ... | ... | ... |

**好处**:
- 前后端开发人员可以独立并行工作
- 接口变更易于追踪和维护
- 降低模块间的耦合度

### 7.3 模块化设计

#### 7.3.1 功能模块的内聚性

**高内聚设计体现**:

1. **业务聚合**: 相关功能集中在同一个模块中
   - 借阅模块: `BorrowController` + `BorrowService` + `BorrowRecord`(Model) + `BorrowRecordMapper`
   - 用户模块: `UserController` + `UserService` + `UserAccount`(Model) + `UserAccountMapper`

2. **单一职责**: 每个类只负责一件事
   - Controller只负责接收请求和返回响应
   - Service只负责业务逻辑
   - Mapper只负责数据访问
   - Model只负责数据结构定义

3. **接口隔离**: Service层可以通过接口抽象，便于替换实现

#### 7.3.2 模块间的耦合度控制

**低耦合策略**:

1. **分层架构**: 严格的层次依赖
   ```
   Controller → Service → Mapper → Database
   (上层依赖下层，下层不知道上层)
   ```

2. **依赖注入**: 通过Spring IoC容器管理依赖关系
   ```java
   @Autowired
   private BookService bookService;
   ```

3. **DTO/VO模式**: 层间数据传输使用专门的对象，避免直接暴露Entity
   - Controller接收DTO (Data Transfer Object)
   - Service内部使用Entity
   - Controller返回VO (View Object)

4. **事件驱动**: 解耦模块间通信（可扩展方向）
   - 借阅成功事件 → 触发通知发送
   - 逾期事件 → 触发罚款计算

5. **异步处理**: 耗时操作异步执行
   - 导出任务: 异步生成，轮询获取结果
   - 邮件/消息发送: 异步队列处理

#### 7.3.3 设计模式应用

**项目中使用的设计模式**:

| 模式 | 应用场景 | 示例 |
|------|----------|------|
| **MVC模式** | 整体架构 | Controller-View-Model分离 |
| **Repository模式** | 数据访问层 | Mapper接口封装数据库操作 |
| **Singleton模式** | 全局配置 | Spring Bean默认单例 |
| **Strategy模式** | 导出格式 | Excel/PDF导出策略 |
| **Template Method** | 基础Controller | 通用CRUD操作模板 |
| **Observer模式** | 事件监听 | 借阅事件通知 |
| **Factory模式** | 对象创建 | 根据用户类型创建不同对象 |
| **Builder模式** | 复杂对象构造 | 查询条件Builder |
