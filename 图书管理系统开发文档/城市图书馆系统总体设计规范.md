# 城市图书馆系统总体设计规范

## 1. 设计概述

### 1.1 系统定位和目标

**系统名称：** 城市图书馆管理系统（CityLibrarySystem）

**系统定位：** 面向城市公共图书馆/高校图书馆的现代化、智能化图书管理平台，采用前后端分离架构，支持多角色（读者/管理员/超级管理员）协同工作。

**设计目标：**
- **高可用性**：核心业务7×24小时稳定运行，目标可用性 99.9%
- **高性能**：接口响应时间 P99 < 500ms，支持并发用户数 500+
- **安全性**：符合等级保护2.0要求，数据加密传输和存储
- **可扩展性**：模块化设计，支持功能水平扩展和垂直扩展
- **易维护性**：清晰的代码结构、完善的文档、自动化测试覆盖

### 1.2 设计理念

| 理念 | 说明 | 在本系统的体现 |
|------|------|---------------|
| **简洁（Simple）** | 避免过度设计，保持代码清晰易懂 | 单体架构优先、明确分层 |
| **高效（Efficient）** | 优化性能瓶颈，提升用户体验 | MyBatis-Plus ORM优化、前端懒加载 |
| **安全（Secure）** | 安全左移，纵深防御 | JWT认证 + Spring Security + 参数校验 |
| **可扩展（Scalable）** | 预留扩展点，支持未来演进 | 接口抽象、策略模式、配置化 |

### 1.3 适用范围

本文档适用于：
- 城市图书馆管理系统的架构设计和开发
- 系统维护和功能扩展
- 新成员入职培训和技术交接
- 第三方系统集成对接

## 3. 设计原则

### 3.1 SOLID原则在项目中的应用

#### 3.1.1 单一职责原则（SRP）

**定义：** 一个类应该只有一个引起它变化的原因。

**在本项目的应用：**

```java
// ✅ 正确示例：每个Controller只负责一个领域
@RestController
@RequestMapping("/api/books")
public class BookController {
    // 只负责图书相关的HTTP请求处理
    // 不包含用户认证、借阅逻辑等其他职责
}

@RestController
@RequestMapping("/api/borrow")
public class BorrowController {
    // 只负责借阅相关的HTTP请求处理
}

// ❌ 反例（避免）：一个Controller处理多个不相关的领域
// @RestController
// public class MegaController {  // 违反SRP
//     // 同时处理图书、用户、借阅...混乱且难以维护
// }
```

**实际代码体现：**
- [AuthController](file:///g:/CIFS_Java_web/Javaweb/BookSystem_Over/spring-boot-backend/src/main/java/com/example/demo/controller/AuthController.java)：仅负责认证（登录/注册/Token刷新/验证码）
- [BookController](file:///g:/CIFS_Java_web/Javaweb/BookSystem_Over/spring-boot-backend/src/main/java/com/example/demo/controller/BookController.java)：仅负责图书CRUD
- [BorrowController](file:///g:/CIFS_Java_web/Javaweb/BookSystem_Over/spring-boot-backend/src/main/java/com/example/demo/controller/BorrowController.java)：仅负责借阅流程
- [SeatController](file:///g:/CIFS_Java_web/Javaweb/BookSystem_Over/spring-boot-backend/src/main/java/com/example/demo/controller/SeatController.java)：仅负责座位预约（该Controller较大但职责单一）

#### 3.1.2 开闭原则（OCP）

**定义：** 软件实体应该对扩展开放，对修改关闭。

**在本项目的应用：**

```java
// ✅ 使用策略模式实现通知通道扩展
public interface NotificationChannel {
    void send(Notification notification);
    boolean supports(ChannelType type);
}

@Component
public class InAppNotification implements NotificationChannel {
    @Override
    public void send(Notification notification) { /* 站内通知 */ }
    @Override
    public boolean supports(ChannelType type) { return type == IN_APP; }
}

// 扩展新通道时只需新增实现类，无需修改现有代码
@Component
public class SmsNotification implements NotificationChannel {
    @Override
    public void send(Notification notification) { /* 短信通知 */ }
    @Override
    public boolean supports(ChannelType type) { return type == SMS; }
}
```

**当前系统中的OCP体现：**
- [SecurityConfig](file:///g:/CIFS_Java_web/Javaweb/BookSystem_Over/spring-boot-backend/src/main/java/com/example/demo/config/SecurityConfig.java)：通过配置URL白名单控制权限，新增接口只需添加配置
- [GlobalExceptionHandler](file:///g:/CIFS_Java_web/Javaweb/BookSystem_Over/spring-boot-backend/src/main/java/com/example/demo/exception/GlobalExceptionHandler.java)：新增异常类型只需添加@ExceptionHandler方法
- MyBatis-Plus的BaseMapper：通过继承即可获得CRUD能力，无需重复编写

#### 3.1.3 里氏替换原则（LSP）

**定义：** 子类对象必须能够替换其父类对象。

**在本项目的应用：**
- Service实现类可以替换接口引用
- Mapper接口的实现由MyBatis-Plus动态代理生成
- DTO/VO继承关系合理使用

#### 3.1.4 接口隔离原则（ISP）

**定义：** 客户端不应该依赖它不需要的接口。

**在本项目的应用：**

```java
// ✅ 正确：接口小而精
public interface BorrowService {
    BorrowRecord borrowBook(Long userId, Long bookId);
    BorrowRecord returnBook(Long recordId);
    BorrowRecord renewBook(Long recordId);
    Page<BorrowRecord> getMyBorrows(Long userId, Pageable pageable);
}

// ❌ 避免：大而全的接口
// public interface LibraryService {
//     // 混合了图书、用户、借阅、座位等多个领域的方法
// }
```

**实际体现：**
- 每个Service接口只定义该领域的操作
- Controller只注入需要的Service接口
- 前端API模块按领域拆分（auth.ts, book.ts, borrow.ts等）

#### 3.1.5 依赖倒置原则（DIP）

**定义：** 高层模块不应依赖低层模块，两者都应依赖抽象。

**在本项目的应用：**

```java
// ✅ 正确：依赖抽象（接口）
@Service
public class BorrowServiceImpl implements BorrowService {

    // 依赖接口而非具体实现
    private final BookMapper bookMapper;
    private final UserMapper userMapper;
```

### 3.3 KISS原则（Keep It Simple, Stupid）

**定义：** 保持简单直接，避免不必要的复杂性。

| 多租户 | 当前只有单一图书馆实例 | SaaS化运营时 |

#### 4.1.2 HTTP方法语义

| 方法 | 幂等性 | 安全性 | 语义 | 使用场景 |
|------|--------|--------|------|----------|
| **GET** | 是 | 是 | 获取资源 | 查询列表、详情 |
| **POST** | 否 | 否 | 创建资源/执行动作 | 新增、提交表单、特殊操作 |
| **PUT** | 是 | 否 | 完整替换资源 | 全量更新 |
| **PATCH** | 否 | 否 | 部分更新 | 修改某个字段 |
| **DELETE** | 是 | 否 | 删除资源 | 删除数据 |

**幂等性说明：**
- GET：多次请求结果相同
- PUT：多次请求结果相同（最终状态一致）
- DELETE：多次请求结果相同（资源不存在也视为成功）
- POST：每次请求可能产生不同结果（如创建多条记录）

#### 4.1.3 HTTP状态码使用规范

| 状态码 | 类别 | 含义 | 使用场景 |
|--------|------|------|----------|
| **200** | 成功 | OK | 成功GET/PUT/PATCH请求 |
| **201** | 成功 | Created | 成功POST创建资源 |
| **204** | 成功 | No Content | 成功DELETE（无返回体） |
| **400** | 客户端错误 | Bad Request | 参数校验失败 |
| **401** | 客户端错误 | Unauthorized | 未登录或Token无效 |
| **403** | 客户端错误 | Forbidden | 已登录但无权限 |
| **404** | 客户端错误 | Not Found | 资源不存在 |
| **409** | 客户端错误 | Conflict | 资源冲突（如重复预约） |
| **422** | 客户端错误 | Unprocessable Entity | 语义错误（如业务规则违反） |
| **429** | 客户端错误 | Too Many Requests | 请求过于频繁 |
| **500** | 服务端错误 | Internal Server Error | 服务器内部错误 |
| **502** | 服务端错误 | Bad Gateway | 上游服务不可用 |
| **503** | 服务端错误 | Service Unavailable | 服务暂时不可用 |

**本项目统一响应格式（见4.2节），HTTP状态码主要用于快速判断请求成败。**

```java
// GlobalExceptionHandler 统一处理
@RestControllerAdvice
public class GlobalExceptionHandler {

    /**
     * 参数校验异常
     */
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public Result<Void> handleValidationException(MethodArgumentNotValidException e) {
        List<FieldError> errors = e.getBindingResult().getFieldErrors();
        List<Map<String, String>> errorList = errors.stream()
            .map(err -> Map.of("field", err.getField(), "message", err.getDefaultMessage()))
            .toList();

        return Result.error(400, "参数校验失败", errorList);
    }

    /**
     * 业务异常
     */
    @ExceptionHandler(BusinessException.class)
    public Result<Void> handleBusinessException(BusinessException e) {
        return Result.error(e.getCode(), e.getMessage());
    }

    /**
     * 认证异常
     */
    @ExceptionHandler(AuthenticationException.class)
    public Result<Void> handleAuthenticationException(AuthenticationException e) {
        return Result.error(401, "认证失败：" + e.getMessage());
    }

    /**
     * 权限不足异常
     */
    @ExceptionHandler(AccessDeniedException.class)
    public Result<Void> handleAccessDeniedException(AccessDeniedException e) {
        return Result.error(403, "权限不足");
    }

    /**
     * 资源未找到异常
     */
    @ExceptionHandler(EntityNotFoundException.class)
    public Result<Void> handleEntityNotFound(EntityNotFoundException e) {
        return Result.error(404, e.getMessage());
    }

    /**
     * 全局兜底异常
     */
    @ExceptionHandler(Exception.class)
    public Result<Void> handleException(Exception e) {
        log.error("系统异常", e);  // 记录详细日志
        return Result.error(500, "系统内部错误，请稍后重试");
    }
}
```

## 5. 数据库设计规范

### 5.1 范式要求

#### 5.1.1 第一范式（1NF）：原子性

**定义：** 每列都是不可分割的原子值。

**示例：**
```sql
-- ❌ 违反1NF：作者字段存储多个值
CREATE TABLE bad_book (
    id BIGINT PRIMARY KEY,
    title VARCHAR(200),
    authors VARCHAR(500)  -- 存储 "张三,李四,王五"，无法单独查询
);

-- ✅ 符合1NF：使用关联表
CREATE TABLE book (
    id BIGINT PRIMARY KEY,
    title VARCHAR(200)
);

CREATE TABLE book_author (
    book_id BIGINT NOT NULL,
    author_name VARCHAR(100) NOT NULL,
    PRIMARY KEY (book_id, author_name),
    FOREIGN KEY (book_id) REFERENCES book(id)
);
```

**本项目实践：** 大部分表遵循1NF，对于简单的多值属性（如标签）可适当使用JSON字段存储（MySQL 5.7+支持）。

#### 5.1.2 第二范式（2NF）：非部分依赖

**定义：** 非主键字段必须完全依赖于主键（消除部分函数依赖）。

**示例：**
```sql
-- ❌ 违反2NF：复合主键的部分依赖
CREATE TABLE borrow_detail (
    user_id BIGINT NOT NULL,
    book_id BIGINT NOT NULL,
    user_name VARCHAR(100),    -- 只依赖user_id，不依赖book_id
    book_title VARCHAR(200),   -- 只依赖book_id，不依赖user_id
    PRIMARY KEY (user_id, book_id)
);

-- ✅ 符合2NF：拆分为独立的表
CREATE TABLE user_account (id BIGINT PRIMARY KEY, username VARCHAR(100));
CREATE TABLE book (id BIGINT PRIMARY KEY, title VARCHAR(200));
CREATE TABLE borrow_record (id BIGINT PRIMARY KEY, user_id BIGINT, book_id BIGINT, ...);
```

#### 5.1.3 第三范式（3NF）：非传递依赖

**定义：** 非主键字段不能传递依赖于主键（消除传递函数依赖）。

**示例：**
```sql
-- ❌ 违反3NF：传递依赖
CREATE TABLE book_info (
    id BIGINT PRIMARY KEY,
    title VARCHAR(200),
    category_name VARCHAR(50),   -- 依赖category_id
    category_id BIGINT,           -- 依赖id
    FOREIGN KEY (category_id) REFERENCES category(id)
);

-- ✅ 符合3NF：通过JOIN获取category_name
SELECT b.*, c.name as category_name
FROM book b
LEFT JOIN category c ON b.category_id = c.id;
```

**本项目实践：** 基本遵循3NF，对于读多写少且需要频繁JOIN的场景，可适当冗余以提升查询性能（反范式化）。

#### 5.1.4 反范式化场景（性能优化需要时）

**适用条件：**
- 读操作远多于写操作（读写比 > 10:1）
- JOIN查询成为性能瓶颈
- 冗余字段更新逻辑简单可控

**示例：**
```sql
-- 在borrow_record中冗余book_title（避免频繁JOIN查询借阅列表）
ALTER TABLE borrow_record ADD COLUMN book_title VARCHAR(200);
-- 通过触发器或应用层同步维护此字段
```

#### 5.5.3 FOREIGN KEY约束

```sql
-- 外键参照完整性
CONSTRAINT fk_borrow_user
    FOREIGN KEY (user_id) REFERENCES user_account(id)
    ON DELETE RESTRICT  -- 限制删除（有借阅记录的用户不能删除）
    ON UPDATE CASCADE,  -- 级联更新（用户ID更新时同步更新）

CONSTRAINT fk_borrow_book
    FOREIGN KEY (book_id) REFERENCES book(id)
    ON DELETE RESTRICT
    ON UPDATE CASCADE
```

**外键策略选择：**

| 策略 | 含义 | 适用场景 |
|------|------|----------|
| `RESTRICT` | 限制删除/更新 | 重要关联数据（如借阅记录） |
| `CASCADE` | 级联删除/更新 | 从属数据（如订单明细） |
| `SET NULL` | 设为NULL | 可选关联（如可选的外键） |
| `NO ACTION` | 不采取行动 | 与RESTRICT类似 |

**注意：** 本项目使用MyBatis-Plus，可选择在应用层维护外键关系（不创建物理FOREIGN KEY约束），以提升批量操作性能。但在设计文档中应标注关联关系。
