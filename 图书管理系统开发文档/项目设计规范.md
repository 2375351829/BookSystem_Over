# 城市图书馆管理系统 - 项目设计规范

## 1. 规范概述

### 1.1 目的和范围

本文档定义城市图书馆管理系统（CityLibrarySystem）的编码规范、设计标准和开发约定，旨在：
- **统一代码风格**：确保团队成员编写的代码风格一致
- **提升代码质量**：通过规范化减少Bug、提高可维护性
- **加速团队协作**：新成员可快速理解项目结构和编码习惯
- **降低维护成本**：清晰的规范使代码更易阅读和修改

**适用范围：**
- 后端Java/Spring Boot代码
- 前端Vue3/TypeScript代码
- SQL数据库脚本
- Git工作流和代码审查

### 1.2 强制性等级定义

| 等级 | 关键词 | 含义 | 违规处理 |
|------|--------|------|----------|
| **MUST** | 必须 | 必须遵守，无例外 | Code Review必须打回 |
| **SHOULD** | 强烈建议 | 应该遵守，特殊情况可例外 | 需要合理解释 |
| **MAY** | 可选 | 建议遵循，可根据实际情况调整 | 不强制 |
| **MUST NOT** | 禁止 | 绝对禁止 | Code Review必须打回 |

### 2.4 异常处理规范

#### 2.4.1 异常处理原则

| 原则 | 说明 | 正确做法 | 错误做法 |
|------|------|---------|---------|
| **不吞掉异常** | MUST NOT捕获后不做任何处理 | 记录日志并向上层抛出 | 空`catch`块 |
| **不捕获宽泛异常** | SHOULD NOT捕获Exception或Throwable | 捕获具体异常类型 | `catch(Exception e)` |
| **自定义业务异常** | SHOULD使用自定义异常类 | 抛出BusinessException | 直接抛出RuntimeException |
| **全局统一处理** | MUST使用GlobalExceptionHandler | 在一处统一处理异常 | 每个Controller单独try-catch |
| **友好错误信息** | MUST返回用户友好的提示 | "图书库存不足" | "Error: SQLSTATE[23000]" |

#### 2.4.2 正确的异常处理示例

```java
// ✅ 正确：Service层抛出业务异常
@Service
public class BorrowServiceImpl implements BorrowService {

    @Transactional
    public BorrowRecord borrowBook(Long userId, Long bookId) {
        // 1. 检查图书是否存在
        Book book = bookMapper.selectById(bookId);
        if (book == null) {
            throw new BusinessException(404, "图书不存在");
        }

        // 2. 检查图书是否可借
        if (!book.isAvailable()) {
            throw new BookNotAvailableException(bookId);
        }

        // 3. 检查用户是否有未缴罚款
        if (fineRecordService.hasUnpaidFines(userId)) {
            throw new OverdueFineNotPaidException();
        }

        // 4. 执行借阅操作
        BorrowRecord record = new BorrowRecord();
        record.setUserId(userId);
        record.setBookId(bookId);
        record.setBorrowedAt(LocalDateTime.now());
        record.setDueAt(LocalDateTime.now().plusDays(getMaxBorrowDays(user)));
        borrowMapper.insert(record);

        // 5. 更新图书库存
        book.setStockCount(book.getStockCount() - 1);
        bookMapper.updateById(book);

        return record;
    }
}
```

```java
    @ExceptionHandler(BusinessException.class)
    public Result<Void> handleBusinessException(BusinessException e) {
```

#### 2.5.2 日志级别使用指南

| 级别 | 使用场景 | 示例 | 输出频率 |
|------|---------|------|---------|
| **DEBUG** | 开发调试详细信息 | 方法入参、SQL参数、中间结果 | 仅开发环境 |
| **INFO** | 关键业务节点 | 用户登录、借还操作、定时任务执行 | 生产环境正常输出 |
| **WARN** | 可恢复的异常情况 | 参数校验失败、重试操作、降级处理 | 需要关注但不紧急 |
| **ERROR** | 需要人工介入的错误 | 数据库连接失败、外部调用超时 | 立即告警 |

#### 2.5.3 日志规范要点

#### 3.5.2 Store结构模板

```typescript
// stores/authStore.ts
import { defineStore } from 'pinia'
import { ref, computed } from 'vue'
import { login, logout, refreshToken, getUserInfo } from '@/api/auth'
import type { LoginDTO, UserInfo, TokenPair } from '@/types/auth'

export const useAuthStore = defineStore('auth', () => {
  // ===== State ======
  const token = ref<string | null>(localStorage.getItem('token'))
  const refreshTokenStr = ref<string | null>(localStorage.getItem('refreshToken'))
  const userInfo = ref<UserInfo | null>(null)
  const loading = ref(false)

  // ===== Getters ======
  const isLoggedIn = computed(() => !!token.value && !!userInfo.value)
  const isAdmin = computed(() => userInfo.value?.role === 'ADMIN')
  const username = computed(() => userInfo.value?.username ?? '')

  // ===== Actions ======
  async function loginAction(loginData: LoginDTO) {
    loading.value = true
    try {
      const data: TokenPair = await login(loginData)
      token.value = data.accessToken
      refreshTokenStr.value = data.refreshToken
      localStorage.setItem('token', data.accessToken)
      localStorage.setItem('refreshToken', data.refreshToken)

      // 获取用户信息
      await fetchUserInfo()
      return true
    } finally {
      loading.value = false
    }
  }

  async function fetchUserInfo() {
    try {
      userInfo.value = await getUserInfo()
    } catch (error) {
      console.error('获取用户信息失败', error)
      logoutAction()
    }
  }

  function logoutAction() {
    token.value = null
    refreshTokenStr.value = null
    userInfo.value = null
    localStorage.removeItem('token')
    localStorage.removeItem('refreshToken')
  }

  // 返回state和actions
  return {
    // State
    token,
    refreshToken: refreshTokenStr,
    userInfo,
    loading,
    // Getters
    isLoggedIn,
    isAdmin,
    username,
    // Actions
    loginAction,
    fetchUserInfo,
    logoutAction
  }
}, {
  // 持久化配置（可选）
  persist: {
    key: 'auth-store',
    storage: sessionStorage,
    paths: ['userInfo']  // 只持久化用户信息，不同步token（安全考虑）
  }
})
```

## 4. SQL编码规范

### 4.1 关键字规范

| 规则 | 标准 | 示例 |
|------|------|------|
| **关键字大写** | MUST | SELECT, FROM, WHERE, INSERT, UPDATE, DELETE |
| **函数名大写** | MUST | COUNT(), SUM(), AVG(), MAX(), MIN(), NOW() |
| **表名小写** | MUST | user_account, book, borrow_record |
| **字段名小写** | MUST | username, created_at, book_id |
| **字符串值** | MUST 单引号 | `'active'`, `'John Doe'`, `'2024-01-15'` |
| **数值** | 无引号 | 100, 3.14, -1 |
| **NULL** | 无引号 | IS NULL, IS NOT NULL |

**正确示例：**

```sql
SELECT
    b.id,
    b.title,
    b.author,
    c.name AS category_name,
    b.stock_count,
    b.created_at
FROM book b
LEFT JOIN category c ON b.category_id = c.id
WHERE b.status = 'ACTIVE'
  AND b.stock_count > 0
  AND b.created_at >= '2024-01-01'
ORDER BY b.created_at DESC
LIMIT 10 OFFSET 0;
```

**错误示例：**

```sql
-- ❌ 关键字小写
select Id, Title, Author from Book where Status = 'active';

-- ❌ 字段名使用驼峰
SELECT Id, Title, UserName FROM user_account;

-- ❌ 字符串使用双引号
WHERE status = "ACTIVE";
```

### 4.2 查询规范

#### 4.2.1 明确指定列名

**MUST NOT** 使用`SELECT *`（除COUNT(*)外）：

```sql
-- ✅ 明确指定需要的列
SELECT id, username, nickname, avatar_url, role, status, created_at
FROM user_account
WHERE status = 'ACTIVE'
LIMIT 100;

-- ❌ 避免：SELECT *
SELECT * FROM user_account;  -- 返回不需要的数据，浪费资源
```

#### 4.2.2 使用参数化查询

**MUST** 使用MyBatis-Plus的`#{}`参数化（防SQL注入）：

```java
// ✅ 正确：MyBatis-Plus参数化查询（自动防止SQL注入）
@Select("SELECT * FROM book WHERE title LIKE CONCAT('%', #{keyword}, '%')")
List<Book> searchByKeyword(@Param("keyword") String keyword);

// ✅ 正确：使用QueryWrapper（推荐）
QueryWrapper<Book> wrapper = new QueryWrapper<>();
wrapper.like("title", keyword)
       .eq("category_id", categoryId)
       .ge("stock_count", 1)
       .orderByDesc("created_at");
List<Book> books = bookMapper.selectList(wrapper);

// ❌ 错险：字符串拼接（SQL注入风险）
// String sql = "SELECT * FROM book WHERE title LIKE '%" + keyword + "%'";
```

#### 4.2.3 JOIN规范

**SHOULD** JOIN要有明确的ON条件：

```sql
-- ✅ 正确：显式指定JOIN条件
SELECT
    br.id,
    br.borrowed_at,
    br.due_at,
    u.username AS borrower_name,
    b.title AS book_title
FROM borrow_record br
INNER JOIN user_account u ON br.user_id = u.id
INNER JOIN book b ON br.book_id = b.id
WHERE br.status = 'BORROWED'
  AND br.due_at < NOW()
ORDER BY br.due_at ASC;

-- ❌ 错误：隐式JOIN（旧语法，可读性差）
SELECT *
FROM borrow_record br, user_account u, book b
WHERE br.user_id = u.id
  AND br.book_id = b.id;
```

### 4.3 索引使用规范

| 规则 | 说明 | 示例 |
|------|------|------|
| **WHERE条件字段建索引** | 高频查询条件字段应有索引 | `WHERE status = ?` → `idx_status(status)` |
| **避免索引列上使用函数** | 函数会导致索引失效 | `WHERE YEAR(created_at) = 2024` → 失效 |
| **最左前缀原则** | 复合索引要按顺序使用 | 索引(a,b,c)，查询`WHERE a=? AND b=?`可用 |
| **LIKE前缀匹配** | 只有`'xxx%'`能用到索引 | `LIKE 'keyword%'`可用，`'%keyword%'`不可用 |
| **OR条件慎用索引** | OR两边都有索引才有效 | `WHERE a=1 OR b=2`，a和b都要有索引 |

### 4.4 事务处理规范

```sql
-- ✅ 正确：事务范围尽可能小
START TRANSACTION;

-- 1. 扣减库存
UPDATE book SET stock_count = stock_count - 1, updated_at = NOW()
WHERE id = #{bookId} AND stock_count > 0;

-- 2. 创建借阅记录
INSERT INTO borrow_record (user_id, book_id, borrowed_at, due_at, status)
VALUES (#{userId}, #{bookId}, NOW(), DATE_ADD(NOW(), INTERVAL 30 DAY), 'BORROWED');

COMMIT;

-- ❌ 错误：长事务（持有锁时间过长）
START TRANSACTION;
-- ... 一系列耗时操作（如调用外部API、发送邮件等）...
COMMIT;

-- ✅ 正确：明确指定隔离级别（如需要）
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
START TRANSACTION;
-- ...
COMMIT;
```

```
  │     └── release/v1.2.0              发布分支
  │
  └── tags/v1.0.0                      版本标签
```

**分支命名规范：**

| 分支类型 | 命名格式 | 示例 | 说明 |
|---------|---------|------|------|
| **主分支** | `main` / `master` | `main` | 生产环境代码，受保护 |
| **开发分支** | `develop` | `develop` | 开发集成分支 |
| **功能分支** | `feature/{描述}` | `feature/user-import`, `feature/seats-map` | 新功能开发 |

| **发布分支** | `release/{版本}` | `release/v1.2.0` | 版本发布准备 |

### 5.3 Code Review要求

**MUST** 每个PR/MR必须经过Code Review：

**Review Checklist（参考第6章）：**

| 检查项 | 优先级 | 说明 |
|--------|--------|------|
| 功能正确性 | P0 | 实现是否符合需求？边界情况？ |
| 代码质量 | P0 | 命名清晰？无重复代码？符合规范？ |
| 性能影响 | P1 | 有无明显性能问题？N+1查询？ |
| 安全性 | P0 | SQL注入？XSS？敏感信息泄露？ |
| 测试覆盖 | P1 | 关键路径是否有测试？ |
| 文档更新 | P2 | API变更是否更新文档？ |

**Review流程：**
1. 开发者提交PR，自测通过
2. 指定Reviewer（至少1人）
3. Reviewer在3个工作日内完成Review
4. 根据Review意见修改代码
5. Review通过后合并到develop/main

## 6. 代码审查清单

### 6.1 功能正确性

- [ ] **代码实现了预期功能**：对照需求文档/任务描述确认
- [ ] **边界情况已处理**：空值、零值、溢出、并发
- [ ] **异常情况有处理**：网络超时、数据库断连、第三方服务不可用
- [ ] **数据一致性**：事务边界正确？并发安全？

### 6.2 代码质量

- [ ] **命名清晰有意义**：变量名、方法名、类名语义明确
- [ ] **没有重复代码**：DRY原则，公共逻辑抽取
- [ ] **方法长度适中**：单个方法不超过50行（复杂算法除外）
- [ ] **类职责单一**：SRP原则，一个类只做一件事
- [ ] **没有死代码**：注释掉的代码、未使用的导入、无用变量

### 6.3 性能考虑

- [ ] **没有明显的性能问题**：不必要的循环、大量数据全量加载
- [ ] **数据库查询高效**：使用了索引？避免N+1查询？
- [ ] **没有不必要的循环嵌套**：时间复杂度合理？
- [ ] **前端渲染高效**：列表虚拟滚动？大数据分页？

### 6.4 安全性

- [ ] **没有SQL注入风险**：使用参数化查询？
- [ ] **没有XSS风险**：前端转义用户输入？
- [ ] **敏感信息未硬编码**：密码、密钥使用配置中心？
- [ ] **输入已校验**：前后端都做了参数校验？
- [ ] **权限检查到位**：接口鉴权？数据权限？

### 6.5 测试

- [ ] **关键逻辑有单元测试**：核心Service方法有测试覆盖
- [ ] **测试覆盖主要场景**：正常流程、异常流程、边界值
- [ ] **测试可以独立运行**：不依赖外部服务（Mock）

### 6.6 文档

- [ ] **复杂逻辑有注释**：算法、业务规则的说明
- [ ] **API变更已更新文档**：Swagger注解？README更新？
- [ ] **README已更新**（如有必要）：安装步骤、配置说明？

## 7. 文档维护规范

### 7.1 触发更新的场景

**MUST** 在以下场景更新相关文档：

| 触发场景 | 需更新的文档 |
|---------|------------|
| 新增/修改/删除API接口 | API文档（Swagger/OpenAPI）、总体设计规范 |
| 数据库结构变更（DDL） | 数据库设计文档、ER图 |
| 业务规则调整 | 需求文档、扩展规划文档 |
| 新增功能模块 | 架构设计文档、用户手册 |
| 修复重大Bug | CHANGELOG、Release Notes |
| 安全漏洞修复 | 安全加固文档 |
| 依赖升级 | 部署文档、环境要求 |

### 7.2 更新流程

```
1. 修改代码同时更新相关文档
   ↓
2. 将文档变更纳入同一个Commit/PR
   ↓
3. PR描述中列出文档变更清单
   ↓
4. Review时检查文档与代码的一致性
   ↓
5. 合并后发布新版本文档
```

### 7.3 同步检查机制

| 检查项 | 频率 | 负责人 |
|--------|------|--------|
| 代码-文档一致性检查 | 每次发布前 | Tech Lead |
| API文档自动生成验证 | 每次构建 | CI Pipeline |
| 架构文档年度评审 | 每年1次 | 架构师 |
